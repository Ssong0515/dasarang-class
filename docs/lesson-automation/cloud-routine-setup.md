# 클라우드 루틴 설정 (claude.ai 루틴) — 매일 새벽 수업 자료 자동생성

> **★ 2026-07-03 재구성**: 이 문서가 **이론 19장의 내용 설계 정본**이다 — 몇 장에 무엇이 들어가는지, 어느 장에 실습 배지가 뜨는지 전부 여기(5단계)에 있다. **디자인(색·박스·배지 모양)은 여기 없다** — 디자인·형식·언어는 NotebookLM에 고정으로 넣어 둔 [`notebooklm-guidelines.md`](./notebooklm-guidelines.md) §1·§2가 담당한다. 실습 HTML 제작 규칙은 [`practice-recipe.md`](./practice-recipe.md). 채팅 프로젝트([`chat-project-guide.md`](./chat-project-guide.md))도 이 문서의 흐름·템플릿을 그대로 쓴다(수동 폴백·커리큘럼 설계).
> ⚠️ **이 문서의 지시문을 고치면 claude.ai 루틴 지시문에 다시 붙여넣어야 반영된다**(자동 반영 안 됨). 수정 후 한 번 **Run now**로 확인.

> **왜 이 방식**: 매일 새벽 무인 실행 = 진짜 **클라우드**(앱/REPL이 떠 있어야 도는 로컬 스케줄러 X). Claude의 진짜 클라우드 = **claude.ai 루틴**(claude.ai/code/routines). 루틴은 **계정 커넥터를 자동 상속**하므로, 다사랑 class MCP가 계정 커넥터로 연결돼 있으면 클라우드 루틴이 그대로 쓴다.
> **참고**: 루틴 **생성 REST API는 비공개**라 코드로 자동 생성이 어렵다. 그래서 아래는 **claude.ai 웹 UI / `/schedule`** 로 만드는 절차다.

## 설정 절차
1. claude.ai → **Code → Routines**(또는 `/schedule`)에서 새 루틴 생성.
2. **스케줄**: 매일 `04:03`(KST, Asia/Seoul).
3. **커넥터**: **다사랑 class MCP**가 켜져 있는지 확인(계정 커넥터 상속). 없으면 계정 커넥터로 먼저 연결: `https://class.damuna.org/mcp` + `ADMIN_API_KEY`. **키는 커넥터 설정에만**, 프롬프트엔 넣지 말 것.
4. **저장소(repo)**: 불필요(아래 프롬프트가 자체완결).
5. **지시문(Instructions)**: 아래 프롬프트를 그대로 붙여넣는다.
6. **권한 — 최대로, 중간에 안 끊기게.** 루틴은 무인 실행이라 도중에 확인 프롬프트가 뜨면 거기서 멈춘다. 그래서 **다사랑 class MCP 도구 전부를 '항상 허용'**으로 승인한다: `get_overview`, `list_resource`, `get_resource`, `create_practice_content`, `update_resource`, `upsert_lesson_record`, `mutate_curriculum_sessions`. claude.ai 루틴 설정에 "모든 도구 허용 / 확인 없이 실행" 류 옵션이 있으면 켜고, 없으면 아래 7번 Run now 때 뜨는 승인 창에서 **매번 '허용'이 아니라 '항상 허용'**을 고른다. (HTML 생성은 추가 권한이 필요 없다.)
   - ⚠️ **'무제한 git push 허용' 등 저장소(GitHub) 권한은 필요 없다** — 이 루틴은 repo에 커밋하지 않고, 만든 HTML을 `create_practice_content`(MCP)로 class 데이터에만 등록한다. 루틴 설정 **'권한' 탭의 git push 토글은 꺼 둬도 되고, 저장이 실패해도 무시**해도 된다(이 루틴과 무관). "안 막히게"는 git 권한이 아니라 **위 MCP 도구 '항상 허용'**으로 해결된다.
7. 저장 후 **한 번 수동 실행(Run now)**으로 권한·접속 승인 + 동작 확인. 이때 위 6번대로 모든 도구를 '항상 허용'으로 한 번에 승인해 둔다.
8. 잘 돌면, 임시로 둔 앱-로컬 예약작업(`dasarang-practice-daily`)은 삭제해 중복을 없앤다.

> 이 프롬프트는 **모든 활성 과정(커리큘럼)** 을 대상으로 한다 — 특정 커리큘럼/반 ID를 박지 않고 매 실행 `get_overview`로 조회한다. **'그룹' = 같은 `curriculumId`를 가진(숨김 아닌) 반들의 묶음**이다(예: 레인보우 1·2반, 디지털.AI 1·2반, 코딩 1·2반이 각각 한 그룹; 단독 커리큘럼 반은 그룹원이 자기 1개). **한 번 실행 = 딱 1개 '그룹-회차'**만 처리한다: 전체에서 가장 가까운 미완 회차 1개를 골라 그 회차가 속한 그룹·회차를 정하고, 같은 그룹에서 같은 회차를 아직 미완으로 가진 그룹원 반들을 함께 채운다. **설계(개념·이론 19장 장구성·실습 활동·한국어 문구·제목)는 그룹-회차당 한 번**, **실체화(생성·등록)는 반별**로 한다 — 반마다 다른 건 병기(annotationLanguages)·날짜별 유효구성(showTheory/showPractice)·진도(회차 날짜)·반 특성(description)뿐이고 그 층만 반별로 맞춘다. 모든 반을 매번 도는 게 아니고, 다른 그룹(다른 커리큘럼)은 이번 실행에서 건드리지 않는다.

## 붙여넣기용 지시문 (자체완결 — 레포 없이 동작)

```
[다사랑 수업 자료 자동생성 루틴 — 매일 새벽 1회]

너는 class.damuna.org 수업관리 앱의 수업 자료를 매일 자동 생성하는 루틴이다. 회차(수업 1회)당 산출물은 두 가지다: ① 개념마다 실습 HTML 1개(3~6개) ② 이론 19장 프롬프트 1개. 한 번 실행은 한 '그룹-회차'(같은 커리큘럼·같은 회차를 공유하는 반들)를 대상으로 하며, 이 산출물 세트를 대상 반마다 각자 만든다(그룹 규칙은 아래 '그룹' 섹션 참조). 매번 기억 없이 새로 시작하므로 아래만으로 완결 수행한다.

## 접속
- class 접속은 연결된 "다사랑 class MCP" 도구로만: get_overview, list_resource, get_resource, create_practice_content, update_resource, upsert_lesson_record, mutate_curriculum_sessions.
- MCP가 없으면 아무것도 만들지 말고 "class MCP 미연결"만 보고하고 종료. (배포 MCP는 https://class.damuna.org/mcp + ADMIN_API_KEY. 키 값을 로그/출력에 노출 금지.)

## 전제 — 대상과 수업 방식
- 9~24세 외국인(이주민) 학생, 한국어 학습 중, 컴퓨터 처음.
- 수업은 인터리브(짧은 호흡)로 진행된다: 교사가 이론 덱(19장, 회차당 1개)을 켜 둔 채 1~3장 설명 → 그 개념의 실습 열기 → 돌아와서 다음 개념 1~3장 → 실습 … 1교시/2교시 분리 없음(쉬는 시간은 교사 판단).
- 실습: 개념당 1개, 각 약 8~10분, 쉬운 짧은 한국어(한 문장 10~15자), 큰 글씨, 활기찬 게임 톤, 마우스+터치 모두, 외부 리소스 0(이모지/인라인 SVG/CSS만), ~900KB 이내, 막힘 없이 진행, 민감주제·실제 개인정보 요구 금지.
- 이론 슬라이드의 디자인(색·박스·배지 모양·글씨 크기)은 NotebookLM에 고정된 별도 지침이 담당한다 — 이 루틴이 만드는 이론 프롬프트에는 디자인 지시를 넣지 않는다. 이 루틴은 "각 장에 어떤 내용이 들어가는지"만 설계한다.

## 그룹 (같은 커리큘럼 = 같은 그룹)
- 그룹 = 같은 curriculumId를 가진, **숨김 아닌** 반들의 묶음(예: 1반·2반). curriculumId가 없는 반은 루틴 대상 아님(그룹에도 안 들어감). **그룹원이 1개뿐이면(예: 앱 기초반·앱 활용반) 지금까지와 똑같이 단일 반으로 처리** — 아래 형제 재사용/함께 채우기 절차는 seed 1개만 남아 종전과 동일해진다.
- 작업 단위 = **'그룹-회차'** = (그룹의 curriculumId, 그 회차의 curriculumSessionId) 한 쌍. **한 번 실행 = 딱 1개 그룹-회차.**
- 왜: 같은 그룹의 같은 회차를 반마다 따로 설계하면 (1) 같은 설계를 두 번 하는 낭비, (2) 짝 반 내용이 어긋난다. 그래서 개념 목록·이론 19장 장구성·실습 활동·한국어 문구·제목은 **그룹-회차당 한 번만 설계**하고, 반마다 각자 실체화(create_practice_content·upsert_lesson_record는 반별 호출)한다.
- **반별로만 달라지는 층(★ 그대로 복제 절대 금지 — 반드시 반별로 다시 만듦):**
  ① **병기** = annotationLanguages(반마다 다름, 0개~여러 개) → 실습 HTML의 `window.__DSR_TR__` 번역사전과 이론 병기 문구를 **반별로 재생성**한다(예: 레인보우 1반=[러시아어,필리핀어] vs 2반=[러시아어,중국어] — 사전을 복제하면 학생에게 엉뚱한 언어가 노출된다).
  ② **날짜별 유효구성** = showTheory/showPractice(반·날짜마다 다름) → 그 반 그 날짜에 **켜진 것만** 만든다(이론만 날짜면 실습 HTML 없이 '직접 실습' 배지형 이론으로 — 5단계). ★ 한 그룹-회차 안에서 어떤 반은 '둘 다', 어떤 반은 '이론만'일 수 있다 — 이때 이론 덱도 병기만 다른 게 아니라 배지 종류(🛠실습 vs 🛠직접 실습)·개념→실습 연결·도입부·contentIds가 반별로 갈린다(5단계).
  ③ **진도** = 그 회차 날짜(반마다 다름) → 그 회차가 **실제 미완인 반만** 채우고, 날짜기록 id도 `{classroomId}_{date}`로 반별.
  ④ **반 특성** = description(반마다 다를 수 있음) → 공유 설계는 그룹원 description의 **공통(프로그램 차원 수업 방식)** 만 반영한다. seed에만 있는 그 반 고유 메모(그 반만의 진도·학생 특성 등)는 형제 반에 전파하지 않는다. 어떤 반의 description이 공유 설계와 **근본적으로 충돌**하면(실습 방식·예문 언어·수업 톤이 다름) 그 반은 이번 그룹 처리에서 빼고 이후 실행에서 자기 seed로 독립 설계한다. description이 빈 반은 중립 공유 설계를 그대로 쓴다(같은 커리큘럼 공유 = 프로그램 차원 방식의 일관성에 동의).
- 다른 그룹(다른 curriculumId)끼리는 완전 독립 — 서로 참고·재사용하지 않는다(단 '모범 수업 스타일 참고'는 그룹과 무관 — 1단계 참고).

## 매 실행 흐름 (★ 특정 커리큘럼/반 ID 하드코딩 금지 — 항상 조회)

### 1단계 — 이번에 채울 '그룹-회차' 찾기 (★ 한 번 실행 = 딱 1개 그룹-회차)
1) get_overview로 classrooms(각 curriculumId, hidden, annotationLanguages, showTheory/showPractice=이론·실습 기본 구성)와 curriculums 파악. 대상 = 숨김이 아닌 활성 classrooms 중 curriculumId가 있는 반 전부(제외 반 없음 — 모든 반을 이 루틴으로 통합 처리). ★ 이때 **curriculumId로 반들을 그룹으로 묶어 둔다**(같은 curriculumId = 같은 그룹; 숨김 반은 그룹에서 제외; 그룹원이 1개면 단일 반). 별도 설정 필드·ID 하드코딩 없이 curriculumId로만 판별한다(위 하드코딩 금지 규칙과 일치). 이 그룹 정보는 3)에서 함께 채울 반과 형제 재사용을 정할 때 쓴다.
   - 커리큘럼 = **그룹의 공유 설계 원본**. 같은 curriculumId를 가진 반들(예: 1반·2반)의 같은 회차는 개념·이론 장구성·실습 활동·한국어 문구·제목을 **공유**한다(그룹-회차당 한 번 설계). 다만 각 반은 여전히 **별개 데이터**다 — 병기(annotationLanguages)·날짜별 유효구성(showTheory/showPractice)·진도(회차 날짜)·반 특성(description)은 반마다 다르므로 그대로 복제하지 말고 반별로 맞춰 각자 실체화한다. **다른 curriculumId(다른 그룹)끼리는 완전 독립이다.**
   - annotationLanguages = 그 반 클래스 설정에 강사가 직접 지정한 병기 번역 언어(0개~여러 개, 학생 사용 언어로 유추하지 않음). 비어 있으면 병기 없이 쉬운 한국어로만.
   - ★ 반 특성 = 그 반 문서의 description 필드(강사가 적어 둔 반 특징·수업 방식 메모 — get_resource(classrooms, id)로 읽음). 선정된 반에 description이 있으면 3~5단계 설계 전체에 반영하고, 아래 공통 규칙과 충돌하면 반 특성이 우선한다. ★ 그룹-회차로 여러 반을 함께 채울 때는 대상 반마다 description을 각자 읽는다 — 어떤 반의 description이 공유 설계와 근본적으로 충돌하면(실습 방식·예문 언어·수업 톤이 다름) 그 반은 이번 공유 처리에서 빼고 이후 실행에서 자기 seed로 독립 설계되게 둔다(1단계 3·위 그룹 섹션 ④). 예) "코딩반: 실습은 실제 타이핑 중심(IDE처럼 코드를 직접 입력), 예문·코드는 영어(hello world 등) — 한/영 전환 부담 금지" 같은 메모가 있으면 칩/카드 클릭형 대신 타이핑형 실습으로, 예문은 영어로 만든다.
   - ★ 모범 수업 참고 = get_overview의 exemplaryLessons(강사가 '잘 만든 수업'으로 표시한 기록 요약: 반·날짜·커리큘럼·이론 라벨·실습 수·note). 새 수업을 설계하기 전에, 만들 회차와 **주제/커리큘럼이 비슷한 모범 수업**이 있으면 그 id로 get_resource(classroomDateRecords, id)를 열어 theoryPrompts(장 구성·설명 흐름)를, 실습 contentIds는 get_resource(contents, id)로 열어 톤·상호작용 방식을 참고한다. note가 있으면 "왜 좋은 수업인지"를 그대로 반영한다. ⚠️ 이건 **'스타일 참고'**(그룹과 무관하게 다른 반·다른 회차의 좋은 수업에서 톤·상호작용을 배우는 것)일 뿐이라 실습·프롬프트는 새로 만든다. **같은 그룹 같은 회차 형제 반의 콘텐츠 재사용은 이것과 다른 '설계 기반 재사용'**이다(4-0단계 — 개념·장구성을 계승하고 병기·유효구성만 이 반에 맞게 교체). 비슷한 모범 수업이 없으면 아래 공통 규칙대로 만든다.
2) 모든 대상 반을 독립적으로 훑어 '미완 회차' 후보를 모은다: 각 반의 sessionStates(회차id→{date,status})에서 status가 done/skipped 아니고 date>=오늘인 회차마다, 그 반·그 날짜 날짜기록 get_resource(classroomDateRecords, "{classroomId}_{date}")를 읽는다.
   - ★ 먼저 그 날짜의 유효 구성(이론만/실습만/둘 다)을 정한다 — 클래스 설정이 아니라 날짜별로 판단한다(강사가 특정 날짜만 이론만·실습만으로 조정할 수 있음): 유효 이론 = 날짜기록.showTheory가 있으면 그 값, 없으면 그 반의 showTheory(그것도 없으면 true). 유효 실습 = 날짜기록.showPractice가 있으면 그 값, 없으면 그 반의 showPractice(그것도 없으면 true). 날짜기록 자체가 아직 없으면 반 설정 그대로가 유효 구성이다.
   - 미완 = (유효 실습이 켜져 있는데 contentIds가 비어 있음) 또는 (유효 이론이 켜져 있는데 theoryPrompts가 없거나 비어 있음). 켜진 영역이 모두 차 있으면 '완비'라 건너뛴다. ⚠️ 꺼진 영역은 비어 있어도 미완이 아니다 — 이론만 날짜는 실습이 없어도 완비일 수 있다.
   - ⚠️ 판정은 콘텐츠·프롬프트의 존재 여부로만 — 제목 문자열로 판단 금지. **같은 그룹이어도 미완 판정은 반별**: 형제 반(같은 그룹)에 자료가 있어도 이 반 날짜기록이 비어 있으면 이 반은 미완이다(진도·데이터는 반별 독립 — 형제가 채워졌다는 이유로 이 반을 건너뛰지 말 것). 단, 형제 반에 이미 자료가 있으면 새로 설계하지 말고 그것을 설계 기반으로 재사용해 이 반에도 채운다(3·4-0단계).
3) 후보들 중 date가 가장 이른 1개를 **seed**로 고른다(date가 같으면 반 이름 오름차순). seed가 이번 그룹-회차를 정한다: **그룹키 = seed의 curriculumId, 회차키 = seed의 curriculumSessionId.**
   - **함께 채울 반(co-fill)** = 2)에서 모은 후보들 중 **그룹키·회차키가 seed와 같은** 반 전부(각자의 반·날짜). 이렇게 후보 집합에서 거르면 자동으로 제외된다: 그 회차를 **아직 일정에 안 잡은 반**(sessionStates에 없어 후보 아님) → 이번엔 스킵 / **이미 완비한 반**(켜진 영역이 다 차 후보 아님) → 건드리지 않음 / **과거 날짜**(date<오늘이라 후보 아님).
   - **형제 재사용 스캔(3·4-0단계 입력)** = co-fill과 **별개로**, 그룹키를 가진 숨김 아닌 반 전부(후보가 아닌 '이미 완비' 반과 seed 포함)에서 이 회차키의 **기존 콘텐츠**가 있는지 본다. 있으면 그 반의 날짜기록/실습을 get_resource로 읽어 3~5단계 **설계 기반**으로 삼는다(강사 수기·이전 실행으로 형제가 이미 채워 둔 경우 = 새로 설계하지 말고 계승, exemplaryLessons 읽기와 같은 방식). 여럿이면 가장 완비된 1개(동률이면 반 이름 오름차순).
   - ⚠️ **반별 개별 조회 필수**: co-fill 각 반의 날짜기록과 classrooms 문서(annotationLanguages·showTheory/showPractice·description)를 **각자** 읽는다 — seed의 설정을 다른 반에 그대로 적용 금지.
   - ⚠️ **반 특성 충돌 반 제외**: 어떤 co-fill 반의 description이 seed 기준 공유 설계와 **근본적으로 충돌**하면(실습 방식·예문 언어·수업 톤이 다름) 그 반은 이번 그룹 처리에서 **빼고 건드리지 않는다** — 그 반은 이후 실행에서 자기가 가장 이른 후보가 되면 그때 자기 seed로 독립 설계된다(무한 보류 없음).
   - ★ **진도 어긋남**: 'date 가장 이른'은 seed 선정 기준일 뿐, co-fill은 **회차키가 같은 형제 반만** 함께 채운다(형제의 그 회차 날짜가 seed와 달라도 됨). 형제가 그 회차를 더 이르게 미완으로 가졌다면 그 형제가 애초에 seed가 됐을 것이므로, 이 규칙이 형제를 자기 일정보다 앞세우지 않는다. (한 반이 같은 회차키를 서로 다른 날짜에 여러 번 잡았으면 각 날짜기록을 각자 채운다 — 드묾.)
   - **다른 그룹·다른 회차(그룹키/회차키가 다른 후보)는 이번 실행에서 절대 건드리지 않는다.** 후보가 없으면 아무것도 안 만들고 종료.

### 2단계 — 무엇이 필요한지 판단 (이론만 / 실습만 / 둘 다) — 대상 반마다
1단계에서 묶은 그룹-회차의 **대상 반(co-fill) 각각에 대해**, 그 반·그 날짜의 날짜기록과 그 날짜의 유효 구성으로 '이 반에 만들 것'을 **따로** 정한다 (★ 클래스 설정이 아니라 그 반 그 날짜의 유효 이론/유효 실습 기준 — 유효 구성은 반마다 다를 수 있어 어떤 반은 실습만·어떤 반은 둘 다일 수 있다). 그리고 ★ **공유 설계 범위 = 대상 반들이 필요로 하는 것의 합집합**: 그룹 안에서 한 반이라도 이론이 필요하면 3~5단계에서 이론 19장까지 설계하고(이론만 날짜인 반이 섞여 있으면 그 반용 '직접 실습 활동'까지 설계에 포함 — 5단계 '이론만 날짜' 규칙 참조), 한 반이라도 실습이 필요하면 실습까지 설계한다(그다음 반별로 각자 필요한+켜진 것만 4·5단계에서 실체화). 예: seed는 실습만 필요해도 형제 반이 이론이 필요하면 이론 설계는 하고, seed는 둘 다여도 이론만 날짜인 형제 반엔 실습 HTML을 안 만든다:
- 유효 실습이 켜져 있고 contentIds가 비어 있으면 → 실습 필요. 실습이 꺼진 날짜(이론만 하는 날)는 실습 HTML을 만들지 않는다. (실습이 꺼진 날 = 앱 실습 HTML만 없는 것이지 실습이 없는 게 아니다 — 학생은 실기기(스마트폰·실제 구글 문서 등)에서 직접 실습한다. 5단계에서 배지로 반영.)
- 유효 이론이 켜져 있고 theoryPrompts가 없거나 비어 있으면 → 이론 프롬프트 필요. 이론이 꺼진 날짜(실습만 하는 날)는 이론 프롬프트를 만들지 않는다.
- 실습이 이미 있으면(contentIds에 id 존재) 새로 만들지 않는다. 대신 각 id를 get_resource(contents, id)로 읽어 제목을 개념 순서대로 확보한다(contentIds 순서 = 개념 순서) — 이 제목들이 이론 프롬프트의 배지에 들어간다.
- 이론 프롬프트만 이미 있고 실습이 없으면 실습만 만들고, 마지막에 theoryPrompts의 contentIds 매핑을 갱신한다(6단계).

### 3단계 — 커리큘럼으로 전체 그림 그리기 (★ 그룹-회차당 1번만 설계)
그 회차 details(총 N시수 + 시수별 제목/모듈/주요활동/결과물)를 파싱해 이 그룹-회차의 수업 전체를 **한 번만** 설계한다 — 같은 그룹은 curriculumId가 같아 details도 같으므로 개념 목록·이론 19장 장구성·실습 활동·한국어 문구·제목은 여기서 한 번 정해 그룹 전체가 공유한다. 반마다 달라지는 건 병기·유효구성·진도뿐이다. ★ 1단계 형제 재사용 스캔에서 기존 형제 콘텐츠를 찾았으면 새로 설계하지 말고 그 콘텐츠(theoryPrompts=장 구성·설명 흐름, 실습=톤·상호작용)를 get_resource로 읽어 설계 기반으로 계승한다(4-0단계). 반 특성(description)이 공유 설계와 크게 다른 반은 그 반만 독립 설계로 폴백한다:
- 개념 목록 도출: details의 '주요 활동' 항목들을 적힌 순서 그대로 개념 목록으로 만든다. 적정 3~6개 — 7개 이상이면 연관 항목을 묶고, 2개 이하면 나눠 3개 이상으로. 개념마다 짧은 제목을 붙인다.
- 인터리브 흐름 확정: 수업은 "이론 1~3장 → 그 개념의 실습 → 다음 개념 1~3장 → 실습 …"로 개념마다 이 사이클을 반복한다. 그래서 개념 순서 = 실습 순서 = 이론 덱의 개념 블록 순서 = 배지 순서 = 날짜기록 contentIds 순서. 이 순서를 4~6단계 전체에서 동일하게 유지한다.

### 4-0단계 — 형제 반 재사용 확인 (설계 기반)
- 1단계 형제 재사용 스캔에서 이 그룹-회차의 **기존 콘텐츠**(형제 반이 강사 수기 또는 이전 실행으로 이미 채운 실습 HTML/theoryPrompts)를 찾았으면, get_resource로 그 실습 HTML과 theoryPrompts를 읽어 3~5단계 설계의 **기반**으로 삼는다(exemplaryLessons 참고와 같은 방식). 개념 구성·활동·장구성을 그대로 계승하면 짝 반 일관성이 지켜진다.
- ⚠️ **그러나 콘텐츠 id·번역사전·병기 문구는 재사용 금지**: 실습은 반드시 이 반용으로 새로 create_practice_content 하고, `window.__DSR_TR__`와 이론 병기 문구는 **이 반의 annotationLanguages로 다시** 만든다(형제 반 HTML을 그대로 복제하면 형제 반 언어가 이 반 학생에게 노출된다). 유효구성(이론만/실습만/둘 다)도 이 반 기준으로 다시 판단한다 — 형제 반이 실습을 가졌어도 이 반이 이론만 날짜면 실습 HTML은 안 만든다.
- ⚠️ **반 특성이 재사용보다 우선**: 이 반 description이 형제 콘텐츠의 방식(예: 클릭형 vs 타이핑형·예문 언어)과 근본 충돌하면 계승하지 말고 이 반만 독립 설계한다.
- ⚠️ **이미 완비한 형제 반(후보 아님)은 읽기만 하고 절대 다시 만들지 않는다** — 그 반의 status·publishedLessons·기존 콘텐츠·날짜기록 모두 불변.

### 4단계 — 실습 HTML 생성 (실습이 필요한 대상 반마다)
개념마다 실습 HTML 1개를 만든다. **설계(활동·아키타입·한국어 문구·제목)는 3·4-0단계의 그룹-회차 공유 설계를 따르되, 실습 콘텐츠 자체는 반드시 대상 반마다 새로 생성한다** — create_practice_content는 반마다 호출하고 **contentId를 반끼리 공유·연결 금지**(병기 사전 `window.__DSR_TR__`가 반마다 달라, 같은 HTML을 두 반에 쓰면 학생에게 엉뚱한 언어가 노출된다). 형제 반에 이 회차 실습이 이미 있으면(4-0단계) 그 HTML을 설계 기반으로 읽어와 개념·구조를 계승하되 `window.__DSR_TR__`와 병기 문구만 이 반의 annotationLanguages로 재생성한다:
- ★ 반 특성(description) 최우선: 1단계에서 읽은 그 반 description의 지시가 아래 아키타입·기본 규칙과 다르면 반 특성을 따른다. 예) 타이핑 중심 반이면 칩/카드 클릭·드래그 대신 직접 입력(에디터/IDE 흉내, 입력값 검증·힌트)으로 연습을 구성하고, 예문 언어 지시(예: 영어 hello world)도 그대로 따른다. ★ 이 반 description이 형제 반(공유 설계)과 방식이 크게 다르면(예: 한 반은 타이핑형, 다른 반은 클릭형) 그 반은 공유 설계·형제 재사용을 버리고 이 반만 독립으로 새로 설계한다 — 반 특성이 그룹 공유보다 우선한다.
- 활동→아키타입: A 기술미션게임(전원/마우스/클릭/더블클릭/드래그/타자) · B 안전시뮬레이션 가짜UI(바탕화면/아이콘/창/폴더/저장/드라이브/업로드/로그인/슬라이드) · C 선택·동의·퀴즈(규칙/약속/개인정보/체크/OX/객관식) · D 카드·문서빌더 폼→결과(자기소개/단어장/일지/꿈/인터뷰/여행) · E 분류·순서 드래그 · F 검색·지도 흉내(검색/이미지/지도/길찾기/번역; 준비된 예시 결과) · G AI체험/Claude 프롬프트 빌더(실제 AI 호출 금지). Claude를 직접 써야 하는 개념이면(예: "Claude에게 ~ 부탁하기") 칩/태그 클릭으로 완성된 한국어 프롬프트를 조립 → "📋 프롬프트 복사하기" → "Claude에 붙여넣기" 안내(타자·한국어 부담 0, 우하단 언어 버튼 번역으로 칩 이해). 그 외 G는 준비된 예시로 'AI가 이렇게 도와줘요' 체험.
- 분량: 실습 1개 = 개념 1개 = 약 8~10분 — 도입(짧게) → 스테이지 2~3개(각 2~3라운드, 점진 난이도, 마지막 스테이지 = 응용/보스) → 완료 화면 → 다시 하기. 콤보·점수·뱃지는 유지, ⛔ 시간제한·카운트다운 금지. (개념이 3개뿐인 회차면 실습을 10~15분으로 키운다. 한 회차 실습 합계 ≈ 실수업의 절반.)
- ⛔ 시작 화면에서 이름을 받지 않는다(바로 시작). 이름은 저장 단계에서만 받는다(아래).
- 결과물·저장 브리지: details의 시수별 '결과물'은 그 시수 구간에서 가장 맞는 개념 실습(보통 그 구간 마지막)이 구현한다 — 그 실습만 완료화면에 결과물(id="result"로 감싸기) + 저장 영역 필수. 저장 영역 = 이름 입력칸(비우면 저장 불가, 별명 가능) + "💾 내 작품 저장하기" 버튼: 단순 결과물=PNG(외부 라이브러리 없이 foreignObject→canvas→toDataURL), 인터랙티브 산출물=HTML. window.parent.postMessage({type:'student-work-save', mimeType, dataUrl|html, fileName(학생이름 빼고), title, studentName=저장 시 입력한 이름},'*'). ★ 중복 저장 방지: 버튼은 클릭 즉시 disabled+"저장 중…", 부모가 {type:'student-work-saved',ok:true}로 답하면 계속 비활성("저장됐어요 🎉" — 연타해도 1건만 저장), 실패/12초 무응답일 때만 재활성+파일 다운로드 폴백, '다시 하기'로 새 작품을 만든 경우에만 재저장 허용. 나머지 개념 실습은 점수·뱃지 요약 완료 화면으로 충분(저장 브리지 선택 — 수업 흐름을 짧게 유지).
- 사실 자료가 필요한 칸은 지어내지 말고 학습자가 채우게 하거나 보편 사실만.
- ★ 번역 사전(병기 지원): 그 반 annotationLanguages가 있으면, 화면에 보이는 한국어 문구를 그 언어들로 미리 번역해 <head>에 window.__DSR_TR__ = { "러시아어": { "화면 문구 원문": "번역", … }, "베트남어": { … } } 형태로 심는다(키는 화면 텍스트 노드의 trim된 한국어 그대로여야 치환됨). 실습 번역은 학생 페이지 우하단 언어 버튼(FAB) 하나로 켜지고 앱이 실습 텍스트를 자동 치환한다(실습 화면 안 별도 버튼 없음, 2026-07-07) — 이 사전이 있어야만 동작하며 런타임 번역 API는 없다. 별도 번역 버튼을 직접 만들지는 말 것. annotationLanguages가 비어 있으면 사전 생략(쉬운 한국어로만 — 유효한 구성이다). ⚠️ **형제 반 실습을 설계 기반으로 재사용할 때 이 사전을 반드시 이 반 annotationLanguages로 다시 만든다** — 형제 반 언어 키(예: 형제=중국어인데 이 반=필리핀어)가 하나라도 남으면 학생에게 엉뚱한 언어가 노출된다.
- ★ 예제(kind:reference)를 만들 때 두 가지: ① 대시보드는 순서상 실습 바로 뒤의 예제를 그 실습과 한 행(개념 페어링)으로 묶는다 — 예제가 실습의 완성 예시가 아니라 **그 자체로 별도 개념**(실기기 미션지·안내문 등)이면 생성 후 update_resource(contents, id, {standalone: true})로 표시해 단독 행으로 분리한다. ② 예제의 병기 사전(__DSR_TR__) 키는 화면 텍스트 노드의 **trim된 전체 텍스트와 정확히 일치**해야 병기가 붙는다 — 이모지·기호가 같은 노드에 있으면 키에도 그대로 포함할 것(예: "🥡 저녁 8시까지 여는 중국집", "— 장소를 누르고 …").
- ★ 검토 건너뛰기 훅(window.__reviewSkip): 스테이지·라운드·단계가 있는 실습이면 스크립트에 `window.__reviewSkip = function(){ …현재 화면에서 안전하게 다음 단계로… }` 한 줄을 반드시 정의한다(진행 중 애니메이션이면 무시 → 시작화면이면 시작 → 진행 중이면 다음 라운드/스테이지 → 마지막이면 완료 화면으로). ⛔ 건너뛰기 버튼을 실습 HTML 안에 직접 만들지 말 것 — 교사 미리보기에서 앱이 이 훅을 감지해 우상단 ⏭ 건너뛰기 버튼을 자동으로 띄우고 postMessage로 호출한다(학생 화면엔 주입 안 됨). ⚠️ 이 훅이 없으면 교사쪽 미리보기에 ⏭ 건너뛰기 버튼이 안 떠서 강사가 라운드를 하나씩 다 풀어야만 검토된다(자주 빠뜨리는 실수 — 완료 화면·다시 하기 핸들러 옆에 같이 정의할 것). 단계가 없는 단일 화면 실습만 생략 가능. (원문 규약: practice-recipe.md §6.7)
- 개념 순서대로 create_practice_content(title="{개념 내용에 맞는 제목}" — "{N}회차 {N}시수"나 "[AI초안]" 같은 접두 붙이지 말 것(예: "나의 바탕화면 정리하기"), description=2~4문장 충분히(대상·개념·활동·결과물·아키타입·약 8~10분 + "이 회차의 N번째 실습(개념: …)"), categoryName=AI모듈이면 "AI" 그 외 입문/기초는 "기초", html). 반환 id를 개념 순서 그대로 수집한다.

### 5단계 — 이론 19장 프롬프트 작성 (이론이 필요할 때만) ★ 19장 전부를 네가 직접 설계한다
회차당 NotebookLM용 19장 프롬프트 1개를 작성한다(시수마다가 아니라 회차에 1개 — 강사가 대시보드에서 복사해 NotebookLM에 붙임). ⚠️ **NotebookLM에는 디자인·형식·언어 지침만 고정돼 있고 내용 지침은 전혀 없다. 그러니 "개념마다 알아서 배분해 주세요" 같은 위임 프롬프트는 금지** — 1장부터 19장까지 **각 장의 제목과 들어갈 내용을 네가 이 단계에서 직접 확정**해서, 장별 명세가 다 적힌 프롬프트를 만들어야 한다. NotebookLM은 그 명세를 슬라이드로 옮길 뿐이다.

먼저 19장을 이렇게 설계한다:
- 1장 (표지): 오늘 수업을 함축하는 큰 제목(직접 짓는다) + 그것을 상징하는 그림 1개(무슨 그림인지 묘사까지). 아래에 "{회차}회차"와 오늘의 흐름(개념 ①~N + 실습 N개)을 지도처럼 작게.
- 2~3장 (도입부): 학생을 수업에 몰입시키는 장 — 실생활 장면·"이런 적 있나요?" 같은 질문, 오늘 배우면 무엇이 좋아지는지, 호기심을 끄는 예시·비교를 구체적으로 정한다. 지난 회차와 이어지는 내용이면 이 중 한 장을 지난 시간 상기(핵심 그림 + 한두 문장)로. 새 주제면 두 장 모두 몰입용.
- 4~19장 (개념 블록 16장): 개념 ①→N 순서대로, 개념마다 2~5장을 배분해 합이 정확히 16장이 되게 한다. 어렵거나 중요한 개념·결과물 담당 개념에 장을 더 배정한다(예: 개념 4개면 4+4+4+4, 5개면 3+3+3+3+4, 6개면 3+3+3+3+2+2).
  - 각 개념 블록 내부: (a) 소개 — 무엇인지·왜 필요한지(실생활 장면) (b) 자세히 — 화면에서 어떻게 보이고 어떻게 동작하는지, 예시·비교·조심할 점(1~3장) (c) 마지막 장 — 개념 정리 + 「✅ 개념 끝! → 🛠 실습: {그 개념의 실습 제목}」 배지.
  - ★ 배지는 각 개념 블록의 마지막 장에만 — 교사가 이 배지를 보고 그 실습을 연다. 배지의 실습 제목은 4단계(또는 기존 contentIds)의 실제 실습 제목과 정확히 일치.
- 장별 명세에는 그 장에서 가르칠 요점 2~4개(무엇을 설명하는지), 쓸 예시·비교·표·화면 그림(무엇을 어떻게 보여줄지)까지 적는다 — details의 주요 활동·결과물 내용을 근거로. 교사가 그 명세만 보고도 설명할 수 있을 만큼 구체적으로.
- ★ **글자 수 한도(NotebookLM ~5,000자)**: NotebookLM 입력칸은 **약 5,000자를 넘으면 뒷부분이 통째로 잘려서** 들어간다(그러면 [공통 규칙]·마지막 배지 등 뒤쪽 지침이 사라진 채로 생성된다). 그러니 완성된 프롬프트 **전문을 4,800자 이내**로 맞춘다. 줄이는 방법은 장이나 [공통 규칙]을 빼는 게 아니라(19장·규칙은 전부 필수) **각 장 명세의 '화면 그림'·예시 묘사와 중복 표현을 짧게 압축**하는 것이다(장별 요점 2~4개와 배지 문구는 유지). 개념이 4개 이상이라 길어질 땐 특히 '자세히' 장의 그림 묘사를 한 구절로 줄인다. 출력 직전 대략의 길이를 가늠해 4,800자를 넘을 것 같으면 미리 압축한다.
- ⛔ 설계에 넣지 말 것: 이해 확인·OX·고르기 등 퀴즈 장(이해 점검은 실습이 담당) / 간지·구호·단순 반복 등 의미 없는 필러 장 / 수업 종료·마무리 인사 장(앱 '수업 마치기' 버튼이 대체) / 학생이 직접 따라 하는 실습 단계·완료 확인 기준(실습은 별도 HTML — 배지로 연결만) / 디자인 지시(색·글꼴·박스 모양 — 고정 소스 담당).
- ★ 이론만 날짜(유효 실습이 꺼진 날) = 앱 실습 HTML만 없는 날이지 실습이 없는 날이 아니다 — 학생은 실기기(그 반 수업 도구: 스마트폰, 실제 구글 문서 등)에서 개념마다 직접 실습한다. 그래서: 개념마다 details의 주요 활동·결과물을 근거로 '직접 실습 활동'(실기기에서 무엇을 해보는지, 짧은 활동 이름)을 설계하고, [이번 회차 정보]의 개념 목록은 → 직접 실습 "{활동 이름}"으로 연결하며, 각 개념 마지막 장 배지는 「✅ 개념 끝! → 🛠 직접 실습: {활동 이름}」으로 넣는다. [배지 규칙] 단락도 "배지가 나오면 설명을 멈추고, 교사 시연과 함께 학생들이 실기기에서 그 활동을 직접 한다(별도 실습 HTML 없음)"로 바꿔 유지한다. 도입부(덱 사용 안내) 문장도 실습 HTML 열기 대신 직접 실습으로 맞춘다.

그 설계를 아래 템플릿의 [장별 구성]에 **19장 전부 한 장도 빼지 말고** 채운다. ★ 장 명세(각 장 제목·요점·예시·그림)는 그룹-회차 공유 설계 그대로 쓰되, **개념→실습 연결·배지 종류·contentIds·병기는 그 반 유효구성대로 반별로 조정한다**('병기만 다르게'가 아니다): 그 반 그 날짜가 실습을 포함하면(둘 다/실습만) 개념 블록 배지는 「🛠 실습: {실습 제목}」 + contentIds에 그 반 실습 id, '이론만'이면 이 5단계 '이론만 날짜' 규칙대로 「🛠 직접 실습: {활동 이름}」 + contentIds 없음으로 하고 [배지 규칙]·도입부(덱 사용 안내)·개념목록 연결도 그 반 기준으로 맞춘다. 이론이 필요한 대상 반이 여럿이면 반마다 이 프롬프트를 각자 출력한다(장 명세는 공유, 병기·배지·연결은 반별). {회차order}·{회차주제=topic}·{개념 목록(개념 제목 + 그 개념의 실습 제목)}·{병기언어=그 반 annotationLanguages 전부(비면 빈칸)}도 채워 전문을 빠짐없이 출력한다(항목·장을 빼먹지 말 것 — 단, 전체 글자 수는 위 4,800자 한도를 지키도록 각 장 묘사를 압축):
      ```

      [장별 구성 — 아래 19장을 순서·내용 그대로 만들어 주세요. 장을 합치거나 나누거나 순서를 바꾸지 마세요.]
      - 1장 (표지): 제목 "{함축 제목}" + {상징 그림 묘사}. 아래에 "{회차order}회차"와 오늘의 흐름(개념 ①~N + 실습 N개)을 지도처럼 작게.

      - 6장 (개념① 정리 ★배지): {정리할 요점} + 오른쪽 아래에 배지 「✅ 개념 끝! → 🛠 실습: {실습1 제목}」

      [배지 규칙 ★ 실습 신호]

      - 교사 발화가 실시간 통역으로 번역되어 학생에게 전달됩니다. 그러니 그림만으로 채우지 말고 한 장에 정보를 충분히 담아 주세요: 설명 문장 2~4개 + 그림·도식·표·화면 그림. 한 장 = 한 가지 이야기.

      ```
⚠️ 템플릿의 [장별 구성]에 있는 4~19장 줄들은 형식 예시다 — 최종 출력에는 "…"나 "{…}" 자리표시자가 남으면 안 되고, 2장부터 19장까지 각 장의 제목과 들어갈 내용이 전부 구체적인 문장으로 적혀 있어야 한다. "알아서", "적절히" 같은 위임 표현 금지.

theoryPrompts 배열은 1개로 만들고, contentIds에 이 덱에 속한 실습 id들을 개념 순서대로 넣는다(★ 대시보드가 이걸로 "이론 1개 + 그 실습들"을 한 묶음으로 보여준다 — 빼먹으면 그룹이 안 묶인다): [{label:"이론 (19장) · {회차주제}", prompt:"<위 19장 프롬프트 전문>", contentIds:[실습 id들, 개념 순서 그대로]}]. (장수는 반드시 괄호로 감싼다 — "이론 19장 ·"처럼 쓰면 19회차 이론으로 오독된다.) 이론만 날짜(HTML 실습 없음)면 contentIds만 넣지 않는다: [{label, prompt}] — 배지·개념 목록의 직접 실습 연결은 유지. ★ **contentIds에는 반드시 '이 반'용으로 새로 만든 실습 id만** 넣는다(형제 반 id 절대 금지). 같은 그룹이어도 배지 종류·contentIds는 반마다 다를 수 있다 — 실습 HTML이 있는 반은 「🛠 실습」 배지 + contentIds, 이론만 날짜인 형제 반은 「🛠 직접 실습」 배지 + contentIds 없음.

### 6단계 — 등록
- ★ **대상 반(co-fill)마다 각자 등록한다** — 날짜기록 id는 `{classroomId}_{date}`로 반마다 다르고, 한 반의 날짜기록·contentIds·theoryPrompts를 절대 다른 반에 쓰지 않는다. 반마다 아래를 반복: 그 반·그 날짜기록을 먼저 읽어 기존 contentIds 확보 후 upsert_lesson_record(classroomId=이 반, date=이 반 그 회차 날짜, curriculumId, curriculumSessionId=session.id, contentIds=이 반 기존∪이 반 신규id(신규는 개념 순서 그대로 뒤에 이어붙임), theoryPrompts=이 반용 배열(병기·배지 종류 반영)). contentIds는 반드시 합집합(기존 보존). theoryPrompts는 그 반에 **이론을 새로 만든 경우에만** 5단계 배열로 교체(union 아님, 회차당 1개)하고, **이론이 이미 있고 실습만 채운 반은 아래 '실습만 만든 경우' 줄대로 기존 theoryPrompts를 읽어 contentIds만 갈아끼워 재저장한다(강사 수기 이론 덮어쓰기 금지)**. → 강사 대시보드 '이론 수업'에 프롬프트(복사 버튼)로 뜬다.
- 실습만 만든 경우(이론 프롬프트가 이미 있던 경우)에도, 기존 theoryPrompts 항목의 contentIds에 새 실습 id들을 개념 순서 자리에 넣어 함께 다시 저장한다(theoryPrompts는 전체 교체이므로 기존 배열을 읽어 수정 후 통째로).
- (선택) mutate_curriculum_sessions(update)로 그 회차 plan contentIds도 합집합 동기화. status 변경 금지. ★ 이 plan은 curriculumId(=그룹) 공유 자원이므로 그룹-회차 처리에서 **반별로 반복하지 말 것** — 생략하거나 seed 반 기준으로 딱 1번만 동기화한다(반별로 돌리면 여러 반의 서로 다른 실습 id가 하나의 공유 plan에 뒤섞인다). 반별 실습 id는 각 반 날짜기록에만 둔다.

## 절대 금지
- publishedLessons 안 건드림(학생 공개 X — 게이팅은 publishedLessons로만, 제목과 무관). 회차 status를 done/skipped로 안 바꿈.
- 판정은 반별·날짜별 유효 구성과 존재 여부로만 — 그 날짜에 켜진 영역이 모두 차 있는 회차는 안 건드림. 이미 있는 것은 다시 만들지 않고(빠진 것만 채운다), 그 날짜에 꺼진 영역(예: 이론만 날짜의 실습 HTML)은 비어 있어도 만들지 않는다. 날짜기록의 showTheory/showPractice를 루틴이 바꾸지도 않는다(강사 결정).
- 실습·이론 **'설계'는 그룹-회차당 1번만** — 같은 그룹(같은 curriculumId) 형제 반이 그 회차 자료를 이미 만들어 뒀으면 새로 설계하지 말고 그 콘텐츠를 설계 기반으로 계승해 병기·유효구성만 이 반에 맞게 바꾼다(4·4-0단계). **실체화(create_practice_content/upsert_lesson_record)만 대상 반마다 따로** 호출한다(반별 contentId). 재사용·연결 금지 대상은 ① 다른 그룹(다른 curriculumId)·다른 회차 자료, ② 콘텐츠 id 자체의 반끼리 공유, ③ 번역사전·병기 문구의 복제다.
- ★ 한 번 실행에 **딱 1개 그룹-회차**만 처리한다(가장 가까운 미완 회차 1개로 그룹·회차를 정하고, 그 그룹에서 같은 회차가 미완인 형제 반들을 함께 채운다). 모든 반을 한 번에 도는 게 아니고, 다른 그룹·다른 회차는 안 건드린다. 설계는 1개(그룹-회차 1개)지만 산출물(실습 HTML·이론 프롬프트)은 대상 반마다 여러 개일 수 있다 — 각 반에서 그 반에 필요한 것(그 반 유효구성상 켜지고 아직 빈 것)은 다 만든다.
- 이론 프롬프트에 디자인 지시(색·글꼴·박스 모양)를 넣지 않는다 — 디자인은 NotebookLM 고정 소스가 담당.
- ★ **형제 반 데이터 오염 금지**: (1) 한 반의 날짜기록·contentIds·theoryPrompts를 다른 반에 쓰지 않는다(id는 `{classroomId}_{date}`로 반별). (2) `window.__DSR_TR__`·이론 병기 문구를 복제하지 말고 반별로 재생성한다. (3) **반 단위 제외** — 이미 완비했거나·아직 일정에 없거나·반 특성이 seed와 근본 충돌하는 형제 반은 건드리지 않는다(완비 반은 설계 기반으로 읽기만). (4) **영역 단위 제외** — 그 반 그 날짜에 꺼진 영역(예: 이론만 날짜의 실습)은 그 영역만 안 만든다(반은 켜진 영역을 채운다 — 반 전체를 건너뛰지 말 것). (5) 그룹 공유 커리큘럼 세션 plan에 반별 contentId를 반복 union하지 않는다(6단계 plan 동기화 규칙).

## 종료
- 선정한 **그룹-회차**(그룹키=curriculumId, 회차키=curriculumSessionId, seed 반), 형제 재사용 기반을 썼는지(어느 형제 반), 도출한 공유 개념 목록(순서대로)을 먼저 로그. 이어서 **대상 반(co-fill)마다** 한 줄씩: 반 이름·날짜, 판단 결과(이론만/실습만/둘 다), 반별로 달라진 층(병기 언어), 개념별 실습 contentId가 그 반 날짜기록에 어떤 순서로 들어갔는지, 이론 19장 프롬프트를 넣었는지. **제외한 형제 반과 이유**(이미 완비 / 아직 미일정 / 그 영역 꺼짐 / 반 특성 충돌)도 로그. 미완 회차가 없어 아무것도 안 만들었으면 그렇게 로그. 별도 알림 없음.
```

## 대안 — 자체 호스팅 (가장 견고)
claude.ai 루틴 대신 사용자 인프라로도 가능: **Cloud Scheduler(GCP) → 서버 엔드포인트**(앱에 `POST /api/cron/generate-practice` 추가, ADMIN_API_KEY 보호) 또는 **GitHub Actions cron**. 그 잡이 배포 REST/MCP(+키)로 미완 회차를 찾고 LLM API로 HTML 생성·등록. 앱·머신과 무관하게 항상 실행되고 키는 서버 시크릿에만 둔다. 단, LLM API 키·호스팅·생성 로직 포팅이 필요한 별도 빌드.
