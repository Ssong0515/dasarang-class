# 클라우드 루틴 설정 (claude.ai 루틴) — 매일 새벽 실습 자동생성

> **★ 2026-07-03 인터리브 개편**: 아래 지시문이 크게 바뀌었다 — 실습은 시수당 1개(20분)가 아니라 **개념당 1개(약 8~10분, 회차당 3~6개)**, 이론 프롬프트는 시수당 16장이 아니라 **회차당 19장 1개**(개념 블록 + 개념 끝 배지). ⚠️ **claude.ai 루틴 지시문에 아래 새 버전을 다시 붙여넣어야 반영된다**(자동 반영 안 됨). NotebookLM 고정 소스 2개(디자인/형식)도 [`notebooklm-guidelines.md`](./notebooklm-guidelines.md) §1·§2 새 버전으로 교체할 것.

> **왜 이 방식**: 매일 새벽 무인 실행 = 진짜 **클라우드**(앱/REPL이 떠 있어야 도는 로컬 스케줄러 X). Claude의 진짜 클라우드 = **claude.ai 루틴**(claude.ai/code/routines). 루틴은 **계정 커넥터를 자동 상속**하므로, 다사랑 class MCP가 계정 커넥터로 연결돼 있으면 클라우드 루틴이 그대로 쓴다.
> **참고**: 루틴 **생성 REST API는 비공개**라 코드로 자동 생성이 어렵다. 그래서 아래는 **claude.ai 웹 UI / `/schedule`** 로 만드는 절차다.

## 설정 절차
1. claude.ai → **Code → Routines**(또는 `/schedule`)에서 새 루틴 생성.
2. **스케줄**: 매일 `04:03`(KST, Asia/Seoul).
3. **커넥터**: **다사랑 class MCP**가 켜져 있는지 확인(계정 커넥터 상속). 없으면 계정 커넥터로 먼저 연결: `https://class.damuna.org/mcp` + `ADMIN_API_KEY`. **키는 커넥터 설정에만**, 프롬프트엔 넣지 말 것.
4. **저장소(repo)**: 불필요(아래 프롬프트가 자체완결).
5. **지시문(Instructions)**: 아래 프롬프트를 그대로 붙여넣는다. ⚠️ **이 문서의 지시문을 고치면**(예: theoryPrompts 추가) claude.ai 루틴 지시문에 **다시 붙여넣어야** 반영된다 — 코드로 자동 반영되지 않는다. 수정 후 한 번 **Run now**로 확인.
6. **권한 — 최대로, 중간에 안 끊기게.** 루틴은 무인 실행이라 도중에 확인 프롬프트가 뜨면 거기서 멈춘다. 그래서 **다사랑 class MCP 도구 전부를 '항상 허용'**으로 승인한다: `get_overview`, `list_resource`, `get_resource`, `create_practice_content`, `update_resource`, `upsert_lesson_record`, `mutate_curriculum_sessions`. claude.ai 루틴 설정에 "모든 도구 허용 / 확인 없이 실행" 류 옵션이 있으면 켜고, 없으면 아래 7번 Run now 때 뜨는 승인 창에서 **매번 '허용'이 아니라 '항상 허용'**을 고른다. (HTML 생성은 추가 권한이 필요 없다.)
   - ⚠️ **'무제한 git push 허용' 등 저장소(GitHub) 권한은 필요 없다** — 이 루틴은 repo에 커밋하지 않고, 만든 HTML을 `create_practice_content`(MCP)로 class 데이터에만 등록한다. 루틴 설정 **'권한' 탭의 git push 토글은 꺼 둬도 되고, 저장이 실패해도 무시**해도 된다(이 루틴과 무관). "안 막히게"는 git 권한이 아니라 **위 MCP 도구 '항상 허용'**으로 해결된다.
7. 저장 후 **한 번 수동 실행(Run now)**으로 권한·접속 승인 + 동작 확인. 이때 위 6번대로 모든 도구를 '항상 허용'으로 한 번에 승인해 둔다.
8. 잘 돌면, 임시로 둔 앱-로컬 예약작업(`dasarang-practice-daily`)은 삭제해 중복을 없앤다.

> 이 프롬프트는 **모든 활성 과정(커리큘럼)** 을 대상으로 한다 — 특정 커리큘럼/반 ID를 박지 않고 매 실행 `get_overview`로 조회한다. **한 번 실행에 전체에서 딱 1회차**(모든 반·회차를 통틀어 가장 가까운 빈 회차 1개)만 만든다 — 반마다 1개씩이 아니다.

```
## 대상/분량 (★ 2026-07-03 인터리브 개편)
- 9~24세 외국인(이주민) 학생, 한국어 학습 중, 컴퓨터 처음.
- 수업 방식: 교사가 이론 덱(19장)을 켜 둔 채 1~3장 설명 → 그 개념의 실습 열기 → 다음 개념 1~3장 → 실습 … 순으로 짧게 끊어 진행한다. 1교시/2교시 분리 없음(쉬는 시간은 교사 판단).
- 그래서 이 루틴은 회차당: **개념(3~6개)마다 실습 HTML 1개**(각 약 8~10분) + **이론 19장 프롬프트 1개**를 만든다. (기존 "시수당 실습 1개(20분) + 시수당 16장 프롬프트"는 폐기.)
- 쉬운 짧은 한국어(한 문장 10~15자), 큰 글씨, 활기찬 게임 톤, 마우스+터치 모두, 외부 리소스 0(이모지/인라인 SVG/CSS만), ~900KB 이내, 막힘 없이 진행, 민감주제·실제 개인정보 요구 금지.

## 매 실행 알고리즘 (★ 특정 커리큘럼/반 ID 하드코딩 금지 — 항상 조회. ★ **한 번 실행 = 전체에서 딱 1회차만**)
1) get_overview로 classrooms(각 curriculumId, hidden, **annotationLanguages**)와 curriculums 파악. 대상 = 숨김이 아닌 활성 classrooms 중 curriculumId가 있는 반 **전부**. ⛔ **제외: 이름에 "앱"이 들어가는 반은 대상에서 완전히 뺀다.** 현재 "앱 기초반"(커리큘럼 "스마트폰 앱 기초", curriculumId 66wBtQxW5kkJ4VbVvCMA)·"앱 활용반"(커리큘럼 "스마트폰 앱 활용", curriculumId oapN6nMlYTbBG1wBMn3b) 두 반 — 이 두 반의 실습·이론 프롬프트는 이 루틴이 만들지 않는다(강사가 수동으로 만든다). 아래 2)의 빈 회차 후보에서도 이 두 반은 반드시 제외한다. (커리큘럼은 내용 템플릿일 뿐 — **1반 1회차와 2반 1회차는 서로 다른 수업**이다. 반마다 따로 본다.) ※ `annotationLanguages` = 그 반 클래스 설정에 강사가 직접 지정한 **병기 번역 언어**(0개~여러 개, 학생 사용 언어로 유추하지 않음). 비어 있으면 병기 없이 쉬운 한국어로만.
2) **모든 대상 반을 독립적으로 훑어 '가장 가까운 빈 회차'를 전체에서 딱 1개만 고른다**(★ 반마다 1개씩이 아니라, 모든 반·모든 회차를 통틀어 단 1개):
   - 각 반의 sessionStates(회차id→{date,status})에서 status가 done/skipped 아니고 date>=오늘인 회차를 본다. 회차마다 그 반·그 날짜 날짜기록 get_resource(classroomDateRecords, "{classroomId}_{date}")를 읽어 **contentIds에 콘텐츠가 하나라도 있으면 '채워짐', 비어 있거나 기록이 없으면 '빈 회차'**. ⚠️ 판정은 **콘텐츠 존재 여부(contentIds)** 로만 — 제목 문자열 금지. **다른 반에 있어도 이 반 날짜기록이 비어 있으면 이 반은 '빈 회차'다**(반끼리 독립).
   - 그렇게 모든 반에서 모은 '빈 회차' 후보들 중 **date가 가장 이른 1개**(= 가장 가까운 수업)를 선정한다. date가 같으면 아무 1개(예: 반 이름 오름차순). → **선정된 그 1개(반·날짜·회차)만 처리하고, 다른 반·다른 회차는 이번 실행에서 절대 건드리지 않는다.** 빈 회차가 아무 데도 없으면 아무것도 안 만들고 종료.
3) **선정된 그 1회차에 대해서만** 아래를 수행한다(★ 2026-07-03 인터리브 개편 — 시수 단위가 아니라 **개념 단위**):
   a) **개념 목록 도출**: 그 회차 details(총 N시수 + 시수별 제목/모듈/주요활동/결과물)를 파싱해, 시수 블록들을 **순서대로 합쳐** '주요 활동' 항목들을 개념 목록으로 만든다(1시수 활동들 → 2시수 활동들 순). 적정 **3~6개** — 7개 이상이면 연관 항목을 묶고, 2개 이하면 나눠 3개 이상으로. 개념마다 짧은 제목을 붙인다. 수업은 "이론 1~3장 → 그 개념의 실습 → 다음 개념 1~3장 → 실습 …"로 진행되므로 **개념 순서 = 실습 순서**다.
   b) **개념마다 실습 HTML 1개**를 그 반 전용으로 새로 생성한다(★ **반마다 독립** — 같은 커리큘럼·회차 자료가 다른 반에 이미 있어도 재사용·연결하지 않고 그 반 것으로 새로 만든다). 시수마다 1개가 아니라 **개념마다 1개**(회차당 보통 3~6개):
      - 활동→아키타입: A 기술미션게임(전원/마우스/클릭/더블클릭/드래그/타자) · B 안전시뮬레이션 가짜UI(바탕화면/아이콘/창/폴더/저장/드라이브/업로드/로그인/슬라이드) · C 선택·동의·퀴즈(규칙/약속/개인정보/체크/OX/객관식) · D 카드·문서빌더 폼→결과(자기소개/단어장/일지/꿈/인터뷰/여행) · E 분류·순서 드래그 · F 검색·지도 흉내(검색/이미지/지도/길찾기/번역; 준비된 예시 결과) · G AI체험/Claude 프롬프트 빌더(실제 AI 호출 금지). **Claude를 직접 써야 하는 개념이면**(예: "Claude에게 ~ 부탁하기") 칩/태그 클릭으로 **완성된 한국어 프롬프트**를 조립 → "📋 프롬프트 복사하기" → "Claude에 붙여넣기" 안내(타자·한국어 부담 0, 번역 버튼으로 칩 이해). 그 외 G는 준비된 예시로 'AI가 이렇게 도와줘요' 체험.
      - 분량: **실습 1개 = 개념 1개 = 약 8~10분** — 도입(짧게) → 스테이지 2~3개(각 2~3라운드, 점진 난이도, 마지막 스테이지 = 응용/보스) → 완료 화면 → 다시 하기. 콤보·점수·뱃지는 유지, ⛔ 시간제한·카운트다운 금지. (개념이 3개뿐인 회차면 실습을 10~15분으로 키운다. 한 회차 실습 합계 ≈ 실수업의 절반.)
      - ⛔ **시작 화면에서 이름을 받지 않는다**(바로 시작). 이름은 저장 단계에서만 받는다(아래).
      - **결과물·저장 브리지**: details의 시수별 '결과물'은 그 시수 구간에서 **가장 맞는 개념 실습(보통 그 구간 마지막)**이 구현한다 — 그 실습만 완료화면에 결과물(id="result"로 감싸기) + 저장 영역 필수. 저장 영역 = **이름 입력칸(비우면 저장 불가, 별명 가능) + "💾 내 작품 저장하기" 버튼**: 단순 결과물=PNG(외부 라이브러리 없이 foreignObject→canvas→toDataURL), 인터랙티브 산출물=HTML. window.parent.postMessage({type:'student-work-save', mimeType, dataUrl|html, fileName(학생이름 빼고), title, studentName=저장 시 입력한 이름},'*'). ★ **중복 저장 방지**: 버튼은 클릭 즉시 disabled+"저장 중…", 부모가 {type:'student-work-saved',ok:true}로 답하면 계속 비활성("저장됐어요 🎉" — 연타해도 1건만 저장), 실패/12초 무응답일 때만 재활성+파일 다운로드 폴백, '다시 하기'로 새 작품을 만든 경우에만 재저장 허용. **나머지 개념 실습은 점수·뱃지 요약 완료 화면으로 충분**(저장 브리지 선택 — 수업 흐름을 짧게 유지).
      - 사실 자료가 필요한 칸은 지어내지 말고 학습자가 채우게 하거나 보편 사실만.
   c) **개념 순서대로** create_practice_content(title="{개념 내용에 맞는 제목}" — **"{N}회차 {N}시수"나 "[AI초안]" 같은 접두 붙이지 말 것**(예: "나의 바탕화면 정리하기"), description=2~4문장 충분히(대상·개념·활동·결과물·아키타입·약 8~10분 + **"이 회차의 N번째 실습(개념: …)"**), categoryName=AI모듈이면 "AI" 그 외 입문/기초는 "기초", html). 반환 id를 **개념 순서 그대로** 수집한다 — 이 순서가 교사가 수업 중 여는 순서이고, 이론 덱의 개념 끝 배지와 1:1로 대응한다.
   d) **회차당 NotebookLM 이론 슬라이드용 19장 프롬프트 1개**를 작성한다(시수마다가 아니라 회차에 1개 — 강사가 대시보드에서 복사해 NotebookLM에 붙임). 디자인·형식(§1·§2, **병기 번역 레이아웃·개념 끝 배지 포함**)은 NotebookLM 소스에 이미 고정돼 있다고 전제하고, {회차order}·{회차주제=topic}·{개념 목록(개념 제목 + b~c에서 만든 그 개념의 실습 제목)}·{이어지는 내용 여부(직전 회차와 주제가 이어지면 예 + 지난 회차 주제 한 줄, 아니면 아니오)}·{병기언어=그 반 annotationLanguages 전부(비면 빈칸)}를 채워 아래 19장 템플릿 **전문을 빠짐없이** 출력한다(짧게 줄이지 말 것). ⚠️ 슬라이드엔 학생이 직접 따라 하는 실습 단계·완료 확인 기준을 넣지 말 것 — 순수 이론, 각 개념 끝 배지로 실습 연결만:
      ```
      이번 회차의 이론 슬라이드를 19장으로 만들어 주세요. 덱 1개로 이 수업 전체를 진행합니다(1교시/2교시 구분 없음). 교사는 덱을 켜 둔 채 1~3장 설명 → 그 개념의 실습 열기 → 돌아와서 다음 개념 1~3장 → 실습 … 순서로 짧게 끊어 진행합니다.

      ```
## 절대 금지
- publishedLessons 안 건드림(학생 공개 X — **게이팅은 publishedLessons로만, 제목과 무관**). 회차 status를 done/skipped로 안 바꿈. **판정은 반별·콘텐츠 존재 여부로** — 그 반 날짜기록에 이미 콘텐츠 있는 회차는 안 건드림. **실습은 반마다 새로 생성**한다 — 같은 커리큘럼·회차 자료가 다른 반에 있어도 재사용·연결하지 말 것(반별 독립). ★ **한 번 실행에 전체에서 딱 1회차(가장 가까운 1개)만** 만든다 — 반마다 1개씩이 아니다. 그 1회차의 **모든 개념 실습(개념당 1개, 보통 3~6개) + 이론 19장 프롬프트 1개**는 다 만든다. ★ **이름에 "앱"이 들어가는 반("앱 기초반"·"앱 활용반", 스마트폰 앱 기초/활용 커리큘럼)은 대상·후보에서 제외 — 실습도 이론 프롬프트도 절대 만들지 않는다.**
```
