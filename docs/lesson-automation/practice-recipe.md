# 실습 인터랙티브 HTML 생성 레시피 (자동화용)

> **입력**: 회차 `details` 1건. **출력**: **개념마다** 자체 완결 인터랙티브 HTML 1개(회차당 보통 3~6개).
> **★ 2026-07-03 인터리브 개편**: 수업은 이론 덱(회차당 19장 1개)을 켜 둔 채 "이론 1~3장 → 그 개념의 실습 → 다음 개념 1~3장 → 실습 …"으로 진행한다(1교시/2교시 분리 없음). 그래서 실습은 **시수당 1개(20분)가 아니라 개념당 1개(약 8~10분)** — 개념 목록은 `details`의 시수별 '주요 활동' 항목들을 순서대로 합쳐 만든다(적정 3~6개, 많으면 묶고 적으면 나눔). **개념 순서 = 날짜기록 contentIds 순서 = 이론 덱 「✅ 개념 끝 → 🛠 실습」 배지 순서**. 이론 19장 구성 정본은 [`cloud-routine-setup.md`](./cloud-routine-setup.md) 5단계(디자인·형식·언어는 [`notebooklm-guidelines.md`](./notebooklm-guidelines.md) §1·§2).
> **★ 수업 흐름·역할(2026-07-18 명문화)**: 한 개념의 사이클 = **이론 설명(강사) → 실습(학생) → 예제(같이)**. 각 산출물의 역할이 다르다 — **이론** = 철저한 개념·설명 도구(강사가 설명한다). **실습** = 방금 배운 이론을 튜토리얼처럼 직접 해 보는 연습장 — 타이머와 함께 공개되고, 학생이 스스로 풀며, 즉각적인 피드백과 반복으로 체득한다(**같은 동작 반복은 최대 3회**). **예제** = 실습에서 충분히 연습한 도구 사용법을 실전에서 쓰는 단계 — 교사가 실제 프로그램으로 시연하고 학생은 보고 이해하며 실제 앱에서 바로 **한 번** 따라 한다. 예제는 연습이 아니라 실전('최종 보스')이므로 반복 루프·라운드·점수·다시 하기를 넣지 않는다 — 실습의 반복 메커닉을 예제에 옮기지 말 것.
> 이 문서가 곧 **생성 프롬프트(시스템 지침)** 다. 클라우드 루틴(C)과 강사 수동 생성 양쪽이 이걸 따른다.
> **★ 반 특성(`classrooms.description`)이 이 레시피의 기본값보다 우선한다(2026-07-03)**: 강사가 반 문서 description에 적어 둔 수업 방식 지시(예: "코딩반 — 실습은 실제 타이핑 중심(IDE처럼 직접 입력), 예문·코드는 영어(hello world 등), 한/영 전환 부담 금지")가 있으면, 아래의 시각 우선·타자 최소화·칩/카드 클릭 같은 기본 원칙 대신 그 지시를 따른다. 반 특성이 없으면 이 문서 기본값대로.
> 디자인 언어는 이론(NotebookLM) [`notebooklm-guidelines.md`](./notebooklm-guidelines.md)와 공유해 이론↔실습 톤을 맞춘다.
> **품질 기준선(2026-06-23 상향)**: [`sample-practice-desktop-mission.html`](./sample-practice-desktop-mission.html)(2회차 3시수)·[`sample-practice-folder-builder.html`](./sample-practice-folder-builder.html)(2회차 4시수). 시각 우선·콤보/뱃지·변별/함정·5단계(보스 포함)·적응형의 **새 표준**이다. (기존 `sample-practice-mouse-mission.html`/`promise-card`는 더 단순한 1회차 초기 버전 = 상향 대상.)
> **G·Claude 프롬프트 빌더 기준선(2026-06-25)**: [`sample-practice-claude-prompt-builder.html`](./sample-practice-claude-prompt-builder.html)(11회차 22시수 "Claude에게 쉬운 한국어 부탁하기"). 칩 클릭 → 완성된 한국어 프롬프트 → 복사 버튼의 **표준**([4.5]).
>
> ⚠️ **2026-06-23 실수업 피드백(사용자)**: 1회차 2시수가 20분 만에 끝났고 "너무 짧고 쉬웠다". 학생 편차 큼(잘하는 애~처음인 애), **외국인이라 한국어 텍스트로만 푸는 문제를 어려워함**. → 아래 [0]·[1]의 **시각 우선·난이도 상향·분량 2배·게임성·시간압박 금지**를 반드시 반영.

## 0. 대상 학습자 (정정 2026-06-21)

- **9~24세 외국인(이주민) 학생, 한국어를 배우는 중.** ❌ 어르신·고령 전제 아님.
- 함의:
  - **🎯 시각 우선 / 한국어 텍스트 최소화 (가장 중요 · 2026-06-23 추가).** 문제 지시는 **그림·아이콘·시각 큐**로 한눈에 알 수 있게. "이걸 찾아요" + 큰 그림(이모지/SVG) 한 개. **한국어 문장을 읽어야만 풀리는 문제는 지양**(외국인 학습자가 어려워함). 매칭·찾기·드래그·고르기는 그림↔그림으로. 텍스트는 보조(작은 라벨)일 뿐.
  - **🌐 번역은 앱이 자동 처리(2026-06-25, 2026-07-07 개정).** 실습 본문은 **한국어로** 만든다. 학생 페이지 **우하단 언어 버튼(FAB) 하나**가 실습 병기 번역·교사 방송 자막의 언어를 모두 제어한다 — 학생이 언어를 고르면 앱이 모든 실습 iframe의 텍스트를 자동 치환한다(크롬 페이지 번역은 샌드박스 iframe 안에서 동작하지 않아 앱이 대신 치환한다). **실습 화면 안에는 번역 버튼이 없고**(예전 우상단 🌐 버튼은 FAB와 중복돼 제거, 교사 검토 화면에만 수동 🌐 버튼이 뜸) → **실습 HTML에 별도 번역 버튼을 만들지 말 것**. ⚠️ 단, 번역은 **런타임 API가 아니라 실습 HTML에 미리 심어 둔 번역 사전(`window.__DSR_TR__`)으로 치환**한다 — 실습을 만들 때 반 설정 `annotationLanguages`(병기 언어)용 사전을 `<head>`에 박아야 번역이 동작한다([6.6]). (단, [4.5] Claude 프롬프트 빌더의 **최종 프롬프트는 한국어 그대로** 둔다.)
  - **활기찬 게임 톤 OK** — 점수, **콤보(연속 성공 🔥)**, 레벨/라운드, 별·**뱃지(완주/콤보왕/정확왕/마스터)**, 가벼운 애니메이션. 단 과하지 않게(어지러운 효과 금지).
  - **한국어는 쉽고 짧게**(언어 학습자라서). 한 문장 10~15자, 한 화면 1~2문장. 어려운 말은 풀어 쓰고 그림·이모지를 함께.
  - 글씨는 **크고 또렷하게**(제목 ~32px, 안내 ~20px). 모바일·저사양에서도 읽히게. ("저시력/어르신" 전제는 뺀다.)
  - **편차 큰 반 → 적응형.** 쉬운 바닥(누구나 완주: 무한 재시도, 막힘 없음, 틀려도 안내) + 높은 천장(변별·함정·보스 라운드로 잘하는 학생도 도전). 한 콘텐츠가 초보~능숙 모두 커버.
  - 한국 생활·이주 배경 존중. 종교·정치 등 민감 주제, 특정 국가 비하·고정관념 금지.

## 1. 분량·난이도 — 실습 1개 = 개념 1개 = 약 8~10분 (★ 2026-07-03 개편)

수업 1회(회차)는 이론 덱 19장 + **개념당 실습 1개(3~6개)**를 번갈아 진행한다. 한 회차 실습 합계 ≈ 실수업의 절반. **실습 1개 = 그 개념만 깊게, 약 8~10분** (개념이 3개 이하뿐인 회차면 10~15분으로 키운다). 흐름:

1. **도입**(30초 이내, 짧게 — 회차당 실습이 여러 개라 도입이 길면 흐름이 끊긴다): 이번에 할 것 한 줄 + 바로 시작. ⛔ **시작할 때 이름을 받지 않는다**(2026-07-03 제거 — 이름은 저장 단계에서만 받는다, [6.5]).
2. **본 활동 = 스테이지 2~3개**: 그 **개념 하나**를 쪼갠 단계들. **각 스테이지는 2~3 라운드 + 점진적 난이도**(R1 쉬움 → R2 중간 → R3 변별/함정). 콤보·점수 누적. ★ **같은 동작·같은 유형의 반복은 최대 3회** — 체득에는 충분하고 4회부터는 지루하다(라운드 상한 3이 이 규칙이다). 분량이 더 필요하면 같은 걸 더 돌리지 말고 다음 단계·변형으로 넘어간다.
3. **마지막 스테이지 = 응용/보스**: 앞 라운드를 섞은 응용. 도전감은 **정확도(무실수)·콤보**로 (⛔ 시간제한·카운트다운 금지 — 언어 학습자 스트레스). "한 번도 안 틀리면 골드 👑" 식.
4. **완료 화면**: 점수·최고콤보·**뱃지**·요약. **`details`의 시수별 '결과물'을 담당하는 실습**(그 시수 구간에서 가장 맞는 개념, 보통 마지막)만 결과물 + **저장(앱 브리지, [6.5] — 이름은 여기서 입력, 저장 버튼은 중복 클릭 방지)** 필수 — `id="result"`로 감싼다. 나머지 실습은 요약 완료 화면으로 충분(저장 선택).
5. **다시 하기** 제공(반복 학습).

목표치: **스테이지 2~3개 × 각 2~3 라운드** → 총 5~9 라운드 → **실사용 약 8~10분**. 짧아도 밀도는 유지 — 2026-06-23 피드백(너무 짧고 쉬움)의 해법은 이제 '한 개당 분량 2배'가 아니라 **개념당 실습 여러 개 + 각각 변별·함정으로 난이도 확보**다. 미니게임형은 라운드당 표적·함정 밀도를 올리고, 카드·문서형은 변별(좋은 예/나쁜 예 고르기)을 넣는다.

**난이도 — 시각적 변별·함정으로 올린다(텍스트 난독 아님):**
- 비슷하게 생긴 것 구분(📄/📃, 🖼️/📷/🎨, 🎵/🎧, 📁/📂), 함정 선택지(닫기 ✕ 옆 `_`·`☐`, 즐겨찾기에 🗑️ 등).
- 라운드마다 표적/방해물 수↑, 함정 밀도↑. 단 **항상 진행 가능**(틀려도 막지 않고 힌트, 무한 재시도).
- 적응형: 핵심(R1~R2)은 누구나 완주, 변별 라운드·보스는 잘하는 학생의 천장.

## 2. 골격 (공통 셸) — sample 기반

모든 실습은 같은 셸을 써서 학습 경험을 일관되게 한다:

- **헤더**: 큰 제목 + 한 줄 격려("천천히 해 봐요. 틀려도 괜찮아요!").
- **트래커**(상단 칩): `주요 활동`을 ①②③… 칩으로. 진행하면 `now`→`done`(✅).
- **카드**: 한 화면 한 동작. 스테이지마다 `innerHTML` 스왑.
- **진행표시**: `0 / N`, 점수·별.
- **피드백 줄**: 즉시 칭찬·안내. **정답=초록(`.feedback`), 오답=빨강(`.feedback.no`)**으로 색 구분(2026-06-23 사용자 요청). `good()`은 `no` 제거, `bad()`는 `no` 추가로 자동 토글.
- **큰 버튼**: `다음 ▶` / `결과 보기 🏆`.
- **완료 화면**: 결과물(카드·표·목록) + **💾 내 작품 저장하기**(앱 브리지) + 다시 하기. 결과물은 `id="result"` 컨테이너로 감싸 저장 대상으로 삼는다(버튼은 밖에).

코드 패턴·CSS 토큰은 sample을 그대로 참고(IIFE 바닐라 JS, 외부 의존 0).

⚠️ **드래그 구현 주의(2026-06-23 실수로 한 번 깨짐):** 파일을 `position:fixed`로 들면 트레이가 줄어 **아래 드롭존이 점프**한다. (1) 드는 순간 같은 크기 **placeholder를 끼워** 자리 유지, (2) drop 판정 `over(x,y,zone)`은 **placeholder 제거 전에** 계산할 것(먼저 빼면 좌표가 어긋나 정상 드롭이 "틀린 폴더"로 오판정). 존이 2줄로 줄바꿈될 때 특히. 기준선 `sample-practice-folder-builder.html`의 `runSort` 참고.

## 3. 디자인 토큰 (이론과 공유)

이론 슬라이드와 **동일한 색상 기준**을 따른다:
- 기본 배경: 흰색/아주 연한 색 (`#ffffff`), 글자: 진한 남색 `#1f2330`.
- 제목·강조: 파란색 또는 보라색 계열 (기본 `#3b6fe0`; 모듈별로 바꿔도 됨). 강조색 1개 원칙.
- 실습 안내 박스: 연한 초록색 또는 하늘색.
- 주의 문구: 연한 주황색 또는 빨간색.
- 번역(병기) 박스: 연한 보라색 `#ede9fe`. (`window.__DSR_TR__` 치환 텍스트 — 앱이 적용, HTML에서 별도 구현 불필요.)
- 완료 박스: 연한 초록색 (`#1a7f47`/`#56cf8c`).
- 버튼 이름: 회색 또는 파란색 테두리 박스.
피할 것: 너무 강한 원색 조합, 검은 배경 중심, 복잡한 그라데이션, 장식이 너무 많은 템플릿, 초등학생용처럼 과하게 귀여운 디자인.

- 글꼴: `"Malgun Gothic","맑은 고딕", system-ui`. '굵게'로 핵심만 강조.
- 라운드 카드(반경 ~20px), 부드러운 그림자, 넉넉한 여백.
- 순서는 번호(①②③)·화살표(→)로 시각화. 애니메이션은 가볍게.
- **마우스 + 터치 모두 동작**. `user-select:none`(입력칸은 예외). 인쇄용 `@media print`로 버튼·트래커 숨김.

## 4. 활동 어휘 → 인터랙션 매핑 카탈로그 ★

개념(=`주요 활동` 항목)의 키워드로 아래 **아키타입**을 고른다. 실습 1개 = 개념 1개이므로 여러 아키타입을 길게 이어 붙이지 않는다(연관 활동을 한 개념으로 묶은 경우에만 2개 조합). **시수의 `결과물`은 그 구간 담당 실습의 완료 화면을 결정한다.** 같은 회차 안의 실습들은 **메커닉이 겹치지 않게** 아키타입·인터랙션을 다르게 고른다.

| 코드 | 아키타입 | 트리거 키워드(주요 활동) | 인터랙션 | 결과물 예 |
|---|---|---|---|---|

## 4.5 Claude 프롬프트 빌더 (아키타입 G — 학생이 Claude를 직접 쓰는 시수) ★

**왜**: 지금 일부 G 실습은 "이런 걸 만들어요"만 보여주고 끝나, 학생이 Claude를 켜도 뭘 해야 할지 모른다. 학생은 **타자가 느리고 한국어가 서툴러** 프롬프트를 직접 못 친다. → 실습 HTML에서 **클릭만으로 완성된 한국어 프롬프트**를 만들어 복사하게 한다.

**패턴**(시수 제목에 "Claude에게 ~", "AI에게 ~ 부탁하기" 등 직접 사용이 들어가면 이 형태로):
1. **칩/태그 고르기**: 프롬프트의 빈칸을 **버튼(칩)**으로 만든다. 예) 「무엇을?」(자기소개/일기/메뉴판…), 「어떻게?」(쉬운 한국어로/짧게/높임말로…), 「추가 조건」(이모지 넣어줘/3문장으로…). 칩은 **그림·이모지 + 짧은 한국어 라벨**(시각 우선). 직접 입력이 꼭 필요한 칸(이름·주제 단어)만 작은 입력칸 1~2개.
2. **실시간 미리보기**: 고른 칩으로 **완성된 한국어 프롬프트**를 하단 큰 박스에 실시간 조립해 보여준다(예: "안녕하세요. 제 자기소개를 **쉬운 한국어로 3문장** 써 주세요. 저는 ___ 입니다."). 최종 프롬프트는 **항상 한국어**(Claude가 한국어를 잘 처리하고, 강사·학생이 결과를 검토하기 쉬움).
3. **복사 버튼**: 큰 **`📋 프롬프트 복사하기`** 버튼. `navigator.clipboard.writeText` → 실패 시 임시 `<textarea>` + `document.execCommand('copy')` 폴백(외부 의존 0). 복사되면 "복사됐어요! 이제 Claude에 붙여넣기 하세요 ✅". ⚠️ **복사는 화면 텍스트가 아니라 선택값(데이터)으로 프롬프트를 다시 조립해 복사**한다 — 학생이 우하단 언어 버튼으로 번역을 켜서 미리보기가 번역돼 보여도 **복사본은 항상 한국어**다(앱 번역은 화면 텍스트만 바꿈). ⚠️ **진행(다음)은 복사 성공이 아니라 '프롬프트 완성'으로** 푼다(클립보드는 사용자 제스처·권한에 따라 실패할 수 있음).
4. **안내**: "① 복사하기 → ② Claude 열기 → ③ 붙여넣기(Ctrl+V) → ④ 보내기" 4단계를 그림으로. (선택) `https://claude.ai/new` 새 탭 링크. ⚠️ 자동 전송·실제 호출은 하지 않는다 — 학생이 직접 붙여넣는다.
5. **한국어 부담**: 칩 의미가 어려우면 학생은 **학생 페이지 우하단 언어 버튼**으로 칩·안내를 자기 언어로 본다(앱이 실습 텍스트를 자동 치환). 그래도 **만들어지는 프롬프트는 한국어**다. ⚠️ **번역은 실습 HTML에 심어 둔 번역 사전이 있어야만 동작한다** — 실습을 만들 때 반드시 화면에 나오는 한국어 문구를 미리 번역해 사전으로 박아라([6.6]). 런타임 번역 API는 없다.
6. **게임성·결과물**: 여러 개를 만들어 보는 스테이지/콤보·뱃지로 분량 확보(시작 이름 입력 없음 — 이름은 저장 시). **완성한 프롬프트(들)** 를 결과물로 `id="result"`에 담아 저장(앱 브리지, [6.5]). 단순 텍스트 카드면 PNG로 저장.

> 핵심: **학생은 타자 없이 칩만 누르고 복사**해서 Claude로 가져간다. 실습은 "프롬프트 만드는 법"을 가르치는 안전한 연습장이고, 실제 대화는 Claude에서 한다.

## 5. 한국어·말투 규칙

- 쉬운 초급 한국어, 짧은 문장, 높임말("~해요", "~봐요"). 학습자는 "여러분".
- 격려: "좋아요!", "잘했어요!", "거의 다 왔어요!". 실패해도 막지 않기("괜찮아요, 다시 해 봐요").
- 핵심 명사에 이모지·그림 병행. 필요하면 쉬운 설명을 괄호로.
- 안전: 개인정보·비밀번호·결제는 "조심하기"를 강조. **실제 개인정보 입력을 요구하지 않는다**(이름 정도만, 선택).

## 6. 출력 규약 (자동화)

- **개념마다 HTML 1개**(회차당 보통 3~6개). sample처럼 프래그먼트(`<title>`+`<style>`+마크업+`<script>`)거나 완결 문서거나, **자체 완결**이면 된다.
- 외부 리소스 **0**(폰트·이미지·스크립트 CDN 금지). 이미지는 이모지·인라인 SVG·CSS로.
- 용량 **~900KB 이내**.
- `create_practice_content` 인자:
  - `title`: **개념 내용에 맞는 제목** — 예: `창 열고 닫기 연습`. **`{N}회차 {N}시수` 같은 번호나 `[AI초안]` 같은 접두는 붙이지 않는다.** (학생 비공개는 `publishedLessons` 게이팅으로만 — 제목과 무관.) 이 제목이 이론 덱 「✅ 개념 끝 → 🛠 실습」 배지에 그대로 들어간다.
  - `description`: **2~4문장**, 나중에 **이 설명만 보고도 무슨 수업인지 AI가 알 수 있게** 충분히. 포함: 대상(9~24세 외국인), **이 회차의 몇 번째 실습(개념)인지**, 핵심 활동·단계, 결과물(담당 실습이면), 아키타입(A~G), 대략 분량(약 8~10분). (단순 한 줄 금지.)
  - `html`: 생성물.
  - `categoryName`: 모듈/주제에 맞게 기존 카테고리명 사용(기초 / WorkSpace / 앱스 / 학생 자료 / AI). AI 모듈 → "AI", 그 외 입문 → "기초".
- 저장 후 반환 `id`를 **`upsert_lesson_record`로 그 커리큘럼을 쓰는 각 반의 해당 날짜(session.plannedDate) "날짜별 수업기록"(`classroomDateRecords`)의 `contentIds`에 넣는다** — 강사가 실제 보고 고르는 화면이라 여기에 넣어야 "미리 선택됨"이 된다. `curriculumSessionId`로 회차↔기록 연결. ⚠️ **학생 공개 아님**(`publishedLessons`는 안 건드림 = 게이팅 유지). (선택) `mutate_curriculum_sessions`로 회차 plan `contentIds`도 동기화 가능. **status 변경·공개는 하지 않는다.**
  - `contentIds`는 **교체**되므로(append 아님), 기존 기록을 먼저 읽어 **합집합**으로 넣을 것. ★ **신규 id들은 개념 순서 그대로** 넣는다 — 이 순서가 수업 중 교사가 여는 순서이자 이론 덱 배지 순서다.

## 6.5 결과물 저장 (앱 브리지) ★ 인쇄 대체

실습은 학생 페이지의 **sandbox iframe**(`allow-scripts allow-same-origin`) 안에서 돈다. 완료 화면의 **💾 내 작품 저장하기** 버튼은 결과물을 파일로 만들어 **부모 앱에 postMessage**하면, 앱이 기존 업로드 파이프라인(`POST /api/drive/upload` → 반 Drive 폴더 + `studentPosts` pending)으로 저장한다. 강사가 "학생 작품"에서 **승인**하면 damuna.org 홈페이지에 공개된다. **앱은 `publishedLessons`를 안 건드림(게이팅 유지).**

**저장 UX (★ 2026-07-03 개정 — 이름은 저장 때, 중복 저장 금지):**
- **이름 입력은 저장 단계에서만**: 완료 화면의 저장 영역에 이름 입력칸(별명 가능)을 두고, **비어 있으면 저장할 수 없게** 막는다(버튼 비활성 또는 "이름을 써 주세요" 안내). 시작 화면에서는 이름을 받지 않는다. 입력한 이름은 `studentName`으로 보내고, 결과물 카드에 이름 표시가 필요하면 이때 반영한다.
- **저장 버튼 중복 클릭 방지**: 누르는 즉시 버튼을 `disabled` + "저장 중…"으로 바꾼다. 부모가 `ok:true`로 답하면 **계속 비활성** 상태로 "저장됐어요 🎉"(같은 작품을 또 저장할 수 없음 — 학생이 장난으로 연타해도 1건만 저장). 실패(`ok:false`)이거나 12초 무응답일 때만 다시 활성화하고 폴백(내려받기)을 안내한다. **'다시 하기'로 작품을 새로 만든 경우에만** 저장 버튼을 다시 활성화한다.

**형식 — 결과물 종류로 정한다:**
- **단순 결과물**(카드·체크표·문서·트리 등) → **PNG 이미지**. 홈페이지 쇼케이스 썸네일이 바로 보임. `id="result"` 노드를 foreignObject로 canvas에 그려 `toDataURL('image/png')`.
- **인터랙티브 산출물**(학생이 만든 웹페이지·게임 등) → **HTML 파일**(`mimeType:'text/html'`, `html` 필드). 이미지로 못 담는 경우.

**브리지 메시지 규약**(부모는 `StudentPage`가 이미 수신·업로드 처리):
- 실습 → 부모: `postMessage({ type:'student-work-save', mimeType:'image/png'|'text/html', dataUrl?:string, html?:string, fileName:string, title:string, studentName:string, anonymous?:boolean }, '*')`. `fileName`엔 학생 이름을 넣지 말 것(서버가 `{학생이름}_파일명`으로 접두). `studentName`은 **저장 단계에서 입력받은 이름**(비면 전송하지 않는다).
- 부모 → 실습: `{ type:'student-work-saved', ok:boolean, error?:string, webViewLink?:string }`. ok면 "저장됐어요 🎉", 아니면 안내 + **이미지로 내려받기** 폴백.
- **폴백**: 최상위로 열렸거나(앱 밖) 12초 내 응답 없으면 파일 다운로드로 대체(외부 의존 0). 저장 헬퍼(`SAVE.capture`/`download`)는 샘플 실습 코드 참고(외부 라이브러리 금지).

⚠️ **선행조건**: 자동 저장이 되려면 (1) 앱에 이 브리지 코드가 배포돼 있어야 하고, (2) 그 반 `classrooms.driveFolderId`가 연결돼 있어야 한다(강사: 클래스 관리 > Google Drive 폴더). 둘 중 하나라도 없으면 학생은 폴백(내려받기)으로 저장한다.

## 6.6 번역 사전 (우하단 언어 버튼) ★ 런타임 API 없음

**실습 병기 번역은 앱이 모든 실습 iframe에 자동 주입하는 치환 엔진**이 처리한다(별도 버튼 만들지 말 것). 학생은 **학생 페이지 우하단 언어 버튼(FAB) 하나**로 언어를 고르고, 그 언어가 실습 텍스트·교사 방송 자막에 함께 적용된다(★ 2026-07-07: 예전 실습 우상단 🌐 버튼은 FAB와 중복돼 제거 — 교사 검토 화면에서만 수동 🌐 버튼이 뜬다). 단, 치환은 **실습 HTML에 번역 사전이 심겨 있을 때만 동작한다**. 런타임 번역 API는 없으므로, 실습을 만들 때 **화면에 보이는 한국어 문구를 미리 번역해 사전으로 박아야** 학생이 자기 언어로 볼 수 있다.

**규약**: `<head>` 안에 아래 형태의 전역을 둔다(언어명 → `{원문(다듬은 한국어): 번역}`). 키는 **화면 텍스트 노드의 trim된 한국어 그대로**여야 치환된다.

```html
<script>
window.__DSR_TR__ = {
  "러시아어": { "프롬프트 복사하기": "Скопировать запрос", "이제 클로드에 붙여넣기 하세요": "..." },
  "베트남어": { "프롬프트 복사하기": "Sao chép câu lệnh", "...": "..." }
};
</script>
```

- **어떤 언어를 넣나**: 그 반 **클래스 설정의 병기 언어 전부**(`get_overview`의 반별 `annotationLanguages`, 강사가 직접 지정, 0개~여러 개). 학생 language로 유추하지 않는다. 비어 있으면 사전 생략(그 실습은 번역되지 않음 — 우하단 언어 버튼 자체는 사전과 무관하게 항상 뜬다) — 쉬운 한국어+그림만으로도 풀리게 만든 게 우선이라 괜찮다.
- **무엇을 번역하나**: 칩 라벨·안내·버튼 등 **화면에 보이는 한국어 문구**. 학생이 입력하는 이름/주제, 그리고 **복사되는 최종 프롬프트는 번역하지 않는다**(항상 한국어).
- 사전은 루틴(이 자료를 만드는 단계)이 직접 번역해 채운다. 키가 화면 문구와 한 글자라도 다르면 그 줄은 치환되지 않으니 **실제 렌더되는 문구를 그대로** 키로 쓴다.

## 6.7 완료 신호 (`dasa-practice-done`) ★ 이동 버튼 만들지 말 것

학생 화면 실습칸의 **단계 다시 보기 ◀ ▶**(그 실습 '하나'를 단계별로 앞뒤로 되짚기 — 콘텐츠 사이 이동이 아니다)는 **'다시하기' 완료 화면까지 끝낸 학생에게만** 열린다(2026-07-18, 2026-07-20 정리). 완료 전 학생은 교사가 새로 공개할 때 자동으로 따라간다. **강사의 학생-화면 미리보기도 예외 없이 실습을 끝내야 뜨고**(2026-07-20 — 실제 학생과 똑같이), 강사가 단계만 확인할 땐 대시보드 실습 미리보기 모달의 **상시** ◀ ▶를 쓴다. ('복습' 라벨은 뗐고 화살표 버튼만 남겼다.) 그 완료 판정에 쓰는 신호를 실습이 보낸다.

**규약**: **완료 화면(결과·뱃지 화면)에 도달하는 순간** 아래 한 줄을 실행한다(중복 호출 무해).

```js
try{ parent.postMessage({ type:'dasa-practice-done' }, '*'); }catch(e){}
```

- **모든 실습 필수**(단일 화면 실습도 활동을 마친 시점에 보낸다). 저장([6.5])과는 무관 — 저장하지 않아도 완료 화면에 도달하면 보낸다. 🚀 확장 미션은 완료 이후의 선택 활동이므로 신호를 확장 미션 뒤로 미루지 않는다. ⚠️ 빠뜨리면 그 실습을 다 끝낸 학생도 페이지 이동이 안 열린다.
- 예제(kind:reference)는 표시 전용이라 신호가 필요 없다(앱이 완료 판정에서 제외).
- ⛔ 학생에게 보이는 페이지 이동·건너뛰기 **버튼(UI)** 을 실습 HTML 안에 직접 만들지 말 것 — 학생 페이지 이동 UI는 앱이 관리한다.
- ★ **단계 이동 훅 `window.__reviewNav(dir)` 필수**(2026-07-19 복원, 2026-07-20 학생 복습에도 사용): 대시보드 실습 미리보기 모달의 ◀ ▶(강사·상시)와 학생 화면의 단계 다시 보기 ◀ ▶(완료한 학생) **둘 다** 이 훅을 쓴다 — 콘텐츠 사이가 아니라 **한 실습 안에서** 단계를 앞뒤로 넘긴다. 앱이 부모→iframe `postMessage({type:'dasa-review-nav', dir:+1|-1})`를 보내면 iframe의 브리지가 `window.__reviewNav(dir)`를 호출한다(옛 `window.__reviewSkip`은 dir>0 폴백으로만 남음).
  - `dir>0`(다음/건너뛰기): 현재 대기 단계를 **정상 속도로 실행**해 효과(색·자르기·정렬·애니메이션)를 보여주며 다음 단계로. 라운드 넘김 포함, 완료 화면까지. 인트로에서 첫 ▶는 실습 진입.
  - `dir<0`(이전): 인트로로 리셋한 뒤 애니메이션 없이 이전 단계까지 **빠르게 재생**(reviewFast).
  - 구현: `var reviewFast=false; function afterDelay(ms,fn){ if(reviewFast){fn();} else {setTimeout(fn,ms);} }`로 단계 진행 setTimeout을 감싼다 → 뒤로 갈 땐 무애니 재생, 현재 forward 단계만 정상 속도. `reviewEnter()`(인트로→첫 단계)·한 단계 전진(기존 성공 경로 재사용)·`reviewGoto(n)`·busy 중 forward 무시. **절대 throw 금지, 학생 조작·`dasa-practice-done`에 영향 없음.** 타이핑·드래그 단계는 값·선택을 프로그램으로 채워 같은 성공 경로를 태운다.
  - (참고) 교사 미리보기에는 `window.__DASA_REVIEW__===true`도 주입되지만(수동 🌐 번역 버튼용) 이 값으로 별도 UI를 만들지 말 것.
  - **경계에서 조용히 멈춘다(2026-07-20)**: 앱은 첫/끝 단계에서 ◀ ▶를 회색·비활성으로 바꾸지 않는다 — 부모가 실습의 현재 단계 위치를 모르고, 위치를 보고받는 계약은 (복잡도 때문에) 두지 않기로 했다. 그러니 `__reviewNav`는 **양 끝에서 그냥 no-op**(첫 단계에서 dir<0, 완료 화면에서 dir>0이면 throw·에러 없이 그대로 머무름)이면 되고, 실습이 위치를 부모에 보고할 필요는 없다.
